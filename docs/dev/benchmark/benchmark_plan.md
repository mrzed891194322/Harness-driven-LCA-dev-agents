# PET 水瓶开发回归基准方案

## 1. 状态与目的

本文是规划材料。当前仓库**尚未实现自动 benchmark runner**；本阶段只定义固定案例、期望产物、证据要求和人工评测方法，不修改 Agent、skill、tool、CLI 或 openLCA 数据。

基准的目标是判断一次变更是否破坏了 LCA 工作流的事实保真、模型连接、数值语义、证据链或标准覆盖，而不是追求某个不透明的单一分数。

## 2. 项目现有工作流

项目当前可概括为：

```text
研究输入 → LCA 执行计划 → LCI 设计与导入 → openLCA 模型与计算 → 解释与质量评价
```

对应能力已有以下基础：

- 计划阶段已有 `plan-maker`、计划 spec、模板和自检约束；
- LCI 阶段已有 Flow、Process、Product System 设计规范、Provider linking 规则和导入工具；
- openLCA 控制层已有描述符查询、导入、模型图读取和产品系统计算能力；
- RAG 已区分标准、openLCA 手册、用户文件和用户数据，并返回来源定位；
- 项目质量门禁已明确禁止伪造 UUID、Provider、方法、来源和计算状态。

主要缺口是：尚无固定端到端案例与跨版本运行记录；计划和 LCI 评价偏结构检查；Provider 的语义与地域匹配、单位换算、Product System 实际连接、空 LCIA 结果、解释阶段完整性/敏感性/一致性检查尚未形成统一回归闭环。

## 3. 基准案例

将教学材料中的 PET 水瓶 Case 1/2/3 定义为开发回归案例。研究功能是向德国销售点交付 1,000 个装有 1 L 静水的 PET 瓶。详细执行约束见 [water_bottle_lca_plan.md](water_bottle_lca_plan.md)。

### 3.1 固定案例事实

每个情景均使用：

| 输入 | 数量 |
| :--- | ---: |
| PET 瓶体 | 60 kg |
| HDPE 瓶盖 | 4 kg |
| PP 标签 | 1 kg |
| 水 | 1,000 kg |
| 空瓶组件合计 | 65 kg |
| 灌装后产品合计 | 1,065 kg |

固定运输需求为：

| 情景 | 运输方式与地域 | 需求 |
| :--- | :--- | ---: |
| Case 1 | Europe 卡车 | 79.25 t·km |
| Case 2 | China 铁路 | 325 t·km |
| Case 2 | Europe 卡车 | 2,782 t·km |
| Case 3 | RoW 卡车 | 19.5 t·km |
| Case 3 | Europe 卡车 | 2,782 t·km |
| Case 3 | Global 海运 | 902.655 t·km |

这些数量是案例事实金标。LCIA 数值不是跨数据库版本的事实金标。

### 3.2 来源解释

本地教程位于 `harness/knowledge/inputs/user_file/水瓶案例学习/水瓶案例学习.md`：研究目的与功能单位见第 37–86 行，运输距离和物料质量见第 191–203 行。教程第 197 行表格给 Case 3 的 RoW 卡车距离为 300 km，而第 199 行正文写为 200 km；本基准为保持固定的 19.5 t·km，明确采用表格中的 300 km。该冲突必须保留在来源/决策记录中。

## 4. 建模约定

基准采用可观察中间连接的分解模型，而不是把每个情景压缩成一个黑箱过程：

1. 粒料供应；
2. 塑料组件生产；
3. 灌装；
4. 销售点运输。

三个情景应共享相同的功能、参考流、前景过程结构和 LCIA 设置，只改变已声明的 Provider 地域代理与运输路线。前景中不存在共同产品时不进行额外分配；背景数据库使用其锁定 system model 已包含的分配逻辑。

## 5. 一次有效运行的期望产物

| 阶段 | 必需产物 | 最低验证 |
| :--- | :--- | :--- |
| 计划 | execution plan、待决事项、假设和来源清单 | 功能单位、参考流、边界、方法和人工决策完整 |
| LCI | Flow/Process/Product System 数据、映射报告、换算表 | ID 引用、交换方向、定量参考流、单位和 Provider 均可验证 |
| openLCA 查询 | Provider/方法查询清单，含名称、UUID、地域和查询时间 | 不按名称猜 UUID；同名结果已消歧 |
| 导入 | 每类实体成功/失败计数、输入与输出哈希 | 无静默跳过、重复实体或超范围写入 |
| 模型图 | 三个 Product System 的节点和连接边 | 无孤立前景输入；下游输入确实连接到预期 Provider |
| LCIA | 原始结果、计算设置、日志和贡献分析 | 结果非空、单位与方法一致、三个情景均按同一设置计算 |
| 解释 | 热点、完整性/敏感性/一致性检查、限制和结论 | 结论不超出教学与开发用途 |
| 运行记录 | 填写完整的运行记录和基线差异 | 可由无对话上下文的审核者复现 |

## 6. 评测维度

1. **案例事实保真**：功能单位、物料、距离和 t·km 是否与金标一致。
2. **计划完整性**：目标、受众、边界、排除项、数据质量、方法和待决事项是否明确。
3. **证据可追溯性**：事实、代理值、Provider 和方法是否有来源位置或工具查询记录。
4. **LCI 结构与语义**：Flow、Process、交换方向、单位、参考流和质量平衡是否正确。
5. **Provider 映射**：名称、UUID、地域、system model 和参考产品是否同时匹配。
6. **Product System 连通性**：default Provider 是否转化成实际模型连接，是否存在孤立节点。
7. **计算有效性**：计算是否真正执行，LCIA 结果是否非空、完整且使用锁定方法。
8. **解释质量**：热点、数据缺口、敏感性、一致性、结论与限制是否相互一致。
9. **可复现性**：环境、输入、输出、提示词、模型、RAG build ID 和耗时是否记录。
10. **效率**：在前述质量不下降时比较 token、工具调用、运行时间和人工修改量。

标准覆盖和工程证据等级按 [lca_quality_evaluation.md](lca_quality_evaluation.md) 单独评价，不能用效率或平均分抵消硬性门禁失败。

## 7. 当前 openLCA 环境观察

2026-07-20 通过只读 IPC 查询观察到：

- 当前实体名称属于 `ecoinvent 3.11`，过程 system model 后缀为 `Cutoff, S`；
- 可用方法同时包含 `CML v4.8 2016 no LT` 和 `CML v4.8 2016`，本基准指定前者；
- PET、HDPE、运输和自来水等查询会返回多个同名或近似同名 Provider，描述符名称本身不展示足够的地域信息；
- 教程中的旧名称在 ecoinvent 3.11 中可能已变更，不能复制 ecoinvent 3.1 名称或 UUID。

这些只是规划时环境观察，不是永久配置。每次运行仍须记录数据库版本、system model、完整 Provider 名称、UUID、地域与方法 UUID。尤其要检查：

- 同名 Provider 是否被错误地域消歧；
- Product System 创建后是否真正采用预期 default Provider；
- 模型图是否有预期节点和边；
- 计算成功状态是否伴随非空的类别结果；空结果不得视为成功。

## 8. 失败分类

| 类别 | 示例 | 建议归责层 |
| :--- | :--- | :--- |
| `input_or_scope` | 功能单位缺失、公开用途未声明 | 用户输入/计划 |
| `evidence` | 来源不可回读、事实与代理未区分 | RAG/计划/报告 |
| `case_fact` | 质量或 t·km 错误 | 计划/LCI |
| `entity_mapping` | 伪造 UUID、错误地域、方法错配 | LCI/openLCA 查询 |
| `schema_or_reference` | 悬空 ID、交换方向错误 | LCI/spec/tool |
| `import` | 部分导入、重复实体、无审计 | openLCA tool |
| `graph_connection` | Product System 有节点但 Provider 未连接 | LCI/openLCA tool |
| `calculation` | 空结果、错误参考流、方法未应用 | openLCA tool |
| `interpretation` | 把潜在影响说成绝对后果、隐瞒限制 | Agent/报告 |
| `review_required` | 公开比较或关键方法选择未获专家复核 | 人工治理 |

一个运行可有多个失败标签；必须保留最早可观测症状与最终根因，不能只记录“Agent 失败”。

## 9. 迭代方法

1. 锁定案例版本和运行环境，复制运行记录模板。
2. 先跑未修改版本，建立首个可复现数值基线；未经复核不得称为金标。
3. 对变更版本执行同一任务，按阶段保存原始输出。
4. 先检查硬性门禁，再比较事实、结构、图连接和 LCIA 数值。
5. 将最小失败案例加入回归集，修复责任层并重跑受影响阶段。
6. 只有环境、映射和结果差异均得到解释并经指定复审人批准，才能晋升新基线。

## 10. 未来自动化方向

- 为案例输入、Provider manifest、模型图和 LCIA 结果定义版本化 schema；
- 自动校验物料合计、t·km、参考流、交换方向和 UUID 引用；
- 自动抓取 Product System 图并与预期拓扑比较；
- 将“工具退出成功但结果为空”变成显式失败；
- 对锁定环境执行数值容差回归，并按影响类别保存差异；
- 从来源 manifest 计算关键事实证据覆盖率；
- 生成机器门禁报告和人工审核包，但不自动替代 LCA 专家或关键审查小组。
