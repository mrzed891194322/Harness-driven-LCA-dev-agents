# 通用 LCA Agent 任务模板

复制本模板到新任务后替换 `{...}`。没有资料的字段保留 `unresolved`，不要删除。任务执行过程中必须持续区分事实、推断、代理值和未解决事项。

---

## 任务

为 `{产品/服务/决策}` 制定并执行一项 LCA，完成计划、LCI、openLCA 建模与计算、解释、质量评价和可复现运行记录。

### 1. 研究目的

- 研究问题：`{要回答的问题}`
- 开展原因：`{决策或研究背景}`
- 预期用途：`{内部改进/研究/第三方报告/其他}`
- 预期受众：`{受众}`
- 是否拟支持向公众披露的比较声明：`{否/是/unresolved}`
- 报告类型与格式：`{内部报告/第三方报告/其他}`

若“公开比较声明”是 `是` 或 `unresolved`，在发布比较结论前设置人工门禁，说明适用的系统等效性、敏感性、不确定性和独立关键审查要求。

### 2. 功能单位与参考流

- 产品系统功能/性能特征：`{功能}`
- 功能单位：`{可测量的数量、质量、时间、地点和性能}`
- 参考流：`{为实现功能单位所需的产品/服务数量}`
- 比较系统的功能等效性：`{说明或 unresolved}`
- 归一化规则：`{单位过程 → 参考流 → 功能单位}`

不得只有产品名称而没有功能；不得把功能单位和参考流混为一谈。

### 3. 系统边界

- 时间范围：`{年份/期间}`
- 地理范围：`{地区}`
- 技术代表性：`{技术/市场/生产路线}`
- 纳入的生命周期阶段和单位过程：`{列表}`
- 排除的阶段、过程、输入或输出：`{列表、理由、可能影响}`
- 截断准则：`{质量/能量/环境显著性及阈值}`
- 多功能过程与分配：`{避免分配/系统扩展/物理/其他}`
- 回收或再利用处理：`{方法或不适用}`
- 边界流程图要求：`{格式和粒度}`

### 4. 数据与质量要求

- 前景数据源：`{工厂数据、测量、用户文件等}`
- 背景数据库：`{数据库版本}`
- system model：`{如 Cutoff, S}`
- 数据时间、地理和技术代表性要求：`{要求}`
- 完整性和精度要求：`{要求}`
- 数据缺口与缺失值规则：`{规则}`
- 不确定性信息：`{分布、区间、数据质量体系}`
- 证据定位格式：`{文件:行号/URL/工具查询 manifest}`

所有背景 Provider 必须从当前 openLCA 环境查询。记录完整名称、UUID、地域、参考产品/流、system model、查询条件、查询时间和选择理由；同名候选必须消歧。

### 5. LCIA 与解释

- LCIA 方法及版本：`{方法}`
- 方法 UUID：`{查询后填写}`
- 影响类别、指标和表征模型：`{列表/采用方法完整 profile}`
- 因子或方法来源：`{来源}`
- 可选元素：`{无/归一化/分组/加权}`
- 热点分析：`{要求}`
- 敏感性分析：`{关键参数与变化范围}`
- 不确定性分析：`{方法}`
- 一致性和完整性检查：`{要求}`

不得把 LCIA 相对指标表述为实际损害、阈值超越、安全边际或风险。若使用归一化、分组或加权，保留变换前结果并披露价值选择。

### 6. 必须输出

#### 6.1 计划

- `execution_plan`：目标、范围、过程分解、数据策略、LCIA、解释、验证和审核步骤；
- `todo_or_decisions`：所有人工决策点、缺失数据、冲突和升级条件；
- `evidence_manifest`：关键事实与来源定位。

#### 6.2 LCI 与 openLCA

- Flow、Process、Product System 的结构化文件；
- 物料/能量/运输换算与质量/能量平衡；
- 人类可读映射报告；
- Provider 与方法查询记录；
- 导入前验证和导入结果；
- 导入后实体清单和模型图；
- LCIA 原始结果、计算设置与日志。

#### 6.3 报告与审核包

- 目标与范围、LCI、LCIA、解释、结论、限制和建议；
- 完整性、敏感性、一致性及数据质量检查；
- 基线差异、失败与人工修改记录；
- 可由无对话上下文审核者复核的运行清单。

### 7. 事实状态标签

为每个关键输入、映射、方法选择和结论使用以下状态之一：

| 标签 | 含义 | 最低记录 |
| :--- | :--- | :--- |
| `fact` | 用户或可靠来源明确给出 | 来源与定位 |
| `inference` | 从已知事实推导 | 前提、推导过程与不确定性 |
| `proxy` | 用替代数据代表缺失对象 | 原对象、代理对象、选择理由与预期偏差 |
| `unresolved` | 证据不足或需要人工决定 | 缺口、影响、负责人和阻塞阶段 |

代理值不能伪装成事实；推断不能覆盖用户明确输入；未解决事项不能通过填入看似合理的默认值“消失”。

### 8. 人工决策点

- `{研究用途或公开比较状态}`
- `{功能等效性}`
- `{关键系统边界/截断}`
- `{分配或回收方法}`
- `{地域或技术代理}`
- `{LCIA 类别/方法适用性}`
- `{敏感性与不确定性范围}`
- `{报告、专家审核或关键审查}`

每个决策记录选项、建议、风险、最终决定、决定人和日期。机器无法可靠判断的专业问题保持 `needs_human_review`。

### 9. 禁止事项

- 不得虚构或凭记忆填写 UUID、Provider、地域、数据库版本、LCIA 方法或因子；
- 不得虚构来源、行号、工具输出、导入成功、模型连接或计算状态；
- 不得以文件存在、工具退出码为 0 或 Agent 自述作为唯一完成证据；
- 不得静默修改用户事实、单位、边界、分配或代理数据；
- 不得用一个总分掩盖某个适用 ISO 硬性要求缺失；
- 不得在未满足要求时宣称 ISO 认证、标准合格、已通过关键审查或总体环境优越；
- 不得覆盖或删除范围未确认的 openLCA/工作区对象。

### 10. 完成前验证

在宣称任务完成前，至少验证：

1. 功能单位、参考流和所有数量均可追溯且单位一致；
2. 质量/能量/运输换算及分配守恒通过；
3. 结构化文件满足 schema，ID、方向和定量参考流正确；
4. 所有输入交换的 Provider 已验证，未使用伪造或悬空引用；
5. 导入成功/失败计数与实际 openLCA 实体一致；
6. Product System 模型图包含预期节点和连接；
7. LCIA 方法、参考流、计算设置与结果非空且一致；
8. 完整性、敏感性和一致性检查有证据；
9. 结论没有超出研究用途，限制没有被隐藏；
10. 运行记录包含版本、哈希、提示词、模型、RAG build ID、耗时和人工修改。

最终结论只能是 `pass`、`conditional_pass`、`fail` 或 `needs_human_review`，并附证据与理由。
