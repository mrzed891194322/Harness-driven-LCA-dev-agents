# Harness LCA Agent 成熟度路线图

## 1. 总体目标

形成可由 LLM agent 协作、由确定性工具约束、由证据和评测验证的端到端 LCA Harness。最终系统应能说明每项结论从何而来、如何计算、经过哪些检查、仍有哪些限制，以及何时必须由 LCA 专家介入。

路线图按能力成熟度推进，不绑定季度。阶段不能仅凭文件存在或 agent 自评完成，必须满足退出条件并附证据。

## 2. 设计原则

1. **目标与范围驱动**：所有建模深度、数据要求和验证强度由研究目标与范围决定。
2. **标准要求可追溯**：重要约束必须能定位到标准、用户资料、数据库查询或项目 spec。
3. **确定性优先**：schema、UUID、单位、方向、公式、连接和数值守恒等可编程问题交给工具验证。
4. **迭代而非单向流水线**：LCI、LCIA 和解释发现问题时，能显式返回目标与范围或数据阶段修订。
5. **不确定性可见**：缺失、假设、代理数据、价值选择和模型限制不得被流畅文本掩盖。
6. **高风险人类负责**：公开比较、关键假设、审查结论和无法自动化判定的标准符合性由合格人员负责。
7. **先正确再优化**：准确性和合规性门禁优先于 token、延迟和自动化程度。

## 3. 持续工作流

| 工作流 | 长期目标 | 主要产物 |
| :--- | :--- | :--- |
| W1 规范契约 | 每类产物均有结构、语义和兼容契约 | spec、schema、模板、版本策略 |
| W2 工作流技能 | 每个 LCA 阶段有显式前后置条件、失败分支和迭代路径 | skill、状态转换、恢复说明 |
| W3 行为规则 | 跨任务安全与质量不变量单一来源 | rule、权限与升级约束 |
| W4 确定性工具 | 查询、校验、计算和副作用可预测、可测试、可审计 | 类型化 tool、错误模型、幂等接口 |
| W5 知识与证据 | 关键输入、推断和结果均能回到原始证据 | 来源记录、版本、证据清单、冲突记录 |
| W6 状态与可观测性 | 运行可恢复、步骤可解释、失败可定位 | 状态模型、事件日志、运行清单、审计轨迹 |
| W7 评测与审核 | 每次变更可与固定案例和专家判断比较 | 基准集、指标、回归报告、审核记录 |

## 4. M0：治理基线

### 目标

把当前隐式约定转为清晰的职责、接口、状态和评测设计，为后续自动化提供稳定边界。

### 核心交付物

- 建立 agent、skill、spec、rule、tool 清单，标记所有者、调用关系、重复内容和缺口。
- 为各构件定义版本、引用和弃用规则；消除一项规则多处复制且含义不一致的情况。
- 定义统一任务状态：`not_started`、`running`、`needs_input`、`needs_review`、`failed`、`completed`，并规定完成证据。
- 定义证据记录最小字段：事实/结论 ID、来源域、原文件定位或工具调用、版本、提取值、转换、假设、置信状态、消费者。
- 形成最小基准集设计，至少覆盖计划、LCI、RAG、openLCA 只读查询和典型失败案例。
- 将 `docs/TODO.md` 的有效条目归入本路线图工作流，重复或失效项有明确处置结论。

### 退出条件

- 所有 Harness 构件均进入清单，且职责冲突有决定记录。
- 至少一个现有端到端案例被固化为可重复运行的基线，输入、环境、输出和已知问题均已记录。
- “完成”“需要人工输入”“失败”能由状态和证据区分，而非只由 agent 文本声称。
- [quality-gates.md](quality-gates.md) 的零容忍规则可映射到明确的自动检查或人工检查责任人。

## 5. M1：可验证的计划与 LCI

### 目标

让现有“计划制定—LCI 构建—openLCA 导入”主链具备机器可验证的结构、语义、追溯和回归能力。

### 核心交付物

- 为 execution plan、todo、Flow、Process、Product System 和映射报告建立版本化 schema 或等价机器契约。
- 将 UUID、引用完整性、定量参考流、输入输出方向、单位/流属性、provider、数值类型和命名规则移入确定性校验器。
- 为来源数据、代理值、单位换算、分配、截断和背景数据映射建立证据链；关键值不可只存在于对话历史。
- 工具区分只读与写入能力；写入前输出变更清单，写入后返回结构化结果，并对重复执行定义幂等行为。
- 建立离线金标与负例：正确案例、缺字段、错误单位、悬空 UUID、错误 provider、重复实体、RAG 无证据和导入失败。
- 对计划与 LCI 的 skill 增加显式前置条件、后置条件、重试上限、回退点和人类介入条件。

### 退出条件

- 基准集中的所有结构和引用错误均被确定性检查捕获，零容忍缺陷为零。
- 每个关键数据、假设、背景映射和生成 UUID 都有可回读来源或生成记录。
- 同一固定输入和环境重复运行产生语义等价结果；差异可由运行清单解释。
- openLCA 隔离数据库完成创建、校验、导入、读取验证和清理的集成测试。
- 计划与 LCI 的人工审核材料足以让审核者不依赖 agent 对话复核模型。

## 6. M2：端到端 ISO LCA

### 目标

在 M1 可信主链上补齐 ISO 14040/14044 所覆盖的 LCA 阶段、报告和迭代关系。

### 核心交付物

- 目标与范围契约覆盖预期应用、原因、受众、比较性声明、功能单位、参考流、系统边界、截断、分配、数据质量、LCIA 方法、假设、限制和审查类型。
- LCI 阶段覆盖数据收集、计算、分配、缺失数据处理、有效性检查和范围回修。
- LCIA 阶段记录方法版本、影响类别、表征模型/因子、可选要素、价值选择和相对结果限制。
- 解释阶段识别重大问题，并执行完整性、敏感性和一致性检查；适用时纳入数据质量与不确定性分析。
- 建立“解释发现问题 → 回到范围/LCI/LCIA → 重算 → 重评”的可恢复迭代状态机。
- 生成可追溯报告，区分内部研究、第三方沟通和公开比较性声明的要求。

### 退出条件

- 至少一个代表性案例从目标与范围运行到报告，所有阶段产物、计算参数和版本可复现。
- 完整性、敏感性、一致性和限制检查均产生结构化结论，未解决项会阻止不当发布。
- 报告中的关键数值可回溯到 LCI exchange、LCIA 结果和转换过程。
- 对外表述不把潜在影响说成绝对环境影响，不把权重单一分数包装成无价值选择的科学结论。
- M1 基准集无关键回归，新增 LCIA、解释和报告基准通过门禁。

## 7. M3：专家级可信与关键审查

### 目标

让高风险 LCA 具备系统性数据质量、不确定性、敏感性和独立审查能力，明确自动化边界。

### 核心交付物

- 建立时间、地域、技术代表性，完整性、精度、一致性、来源和不确定性等数据质量记录与评价。
- 支持参数、情景、分配、截断、背景数据和 LCIA 方法的敏感性/不确定性分析，并保存运行矩阵。
- 建立专家审核包：研究范围、方法、数据证据、模型图、结果、限制、异常、agent 决策轨迹和未解决事项。
- 关键审查流程记录审查范围、审查者独立性与能力、意见、响应、修订和最终结论。
- 公开比较性声明触发独立评审小组路径；未满足时系统不得生成“可公开发布”状态。
- 建立对抗案例，覆盖虚构来源、单位陷阱、错误功能等价、错误分配、数据污染和过度结论。

### 退出条件

- 专家盲审能根据审核包重建关键判断，且关键缺陷为零。
- 高影响假设在预设敏感性范围内的结论稳定性已量化；不稳定时报告明确限制或返回修订。
- 公开比较案例经过规定的独立评审流程，所有意见和响应可追溯。
- 对抗基准不会被提示注入或流畅但无证据的内容绕过硬门禁。

## 8. M4：持续优化

### 目标

在质量门禁不退化的前提下，持续优化模型、工作流、上下文、工具、成本和延迟。

### 核心交付物

- 建立版本化评测看板，按案例、LCA 阶段、错误类型、模型和工具版本对比。
- 对 prompt、skill、agent 编排、检索和模型变化运行相同基准并记录统计差异。
- 建立失败聚类和根因回流：失败必须落到 spec、skill、rule、tool、knowledge 或 eval 的明确改进项。
- 支持模型/供应商替换，但不降低证据格式、工具契约和发布门禁。
- 优化 token、延迟和人工工作量，并报告质量—成本权衡。

### 退出条件

- 每次 Harness 发布均附基准差异、已知限制和回退方案。
- 连续多个发布周期无关键回归，非关键指标阈值按基线证据逐步收紧。
- 成本或延迟优化未绕开必须的检索、确定性校验或专家审核。
- 生产中发现的新错误进入基准集并能在修复后稳定复现。

## 9. 阶段推进规则

- 阶段可并行准备，但不能跳过依赖退出条件。例如可预研 LCIA skill，但在 M1 引用与数值验证不稳定时不能宣称 M2 完成。
- 每个交付物使用 `planned`、`in_progress`、`blocked`、`verified` 四种状态；只有 `verified` 计入退出条件。
- `verified` 必须附证据路径或运行 ID，并注明验证环境与日期。
- 成熟度变更需写入 [decisions.md](decisions.md)，说明满足的退出条件、保留风险和回退条件。
- 标准、数据库、LCIA 方法或 openLCA 主版本变化时，受影响阶段重新进入验证，不沿用未经评估的旧结论。
