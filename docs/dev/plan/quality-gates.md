# Harness LCA Agent 质量门禁

## 1. 门禁目标

质量门禁用于回答三个问题：变更是否破坏既有能力，产物是否达到声明的可信级别，以及何时必须停止自动执行并交给人类。

门禁结果只有：

- `pass`：本风险级别的必需检查全部通过。
- `conditional_pass`：自动检查通过，但只能在记录的限制内使用；不能用于更高风险发布。
- `fail`：存在零容忍问题、必需检查失败或证据不足。
- `needs_human_review`：机器无法可靠判定或规则明确要求专家负责。

不得把“工具成功退出”“文件已生成”或“agent 自称完成”单独视为通过。

## 2. 零容忍项

以下问题在任何成熟度和风险级别均导致 `fail`：

- 伪造标准条款、文献、用户数据、openLCA 实体、UUID、provider、LCIA 方法或工具结果。
- 关键数据、假设、换算、分配或背景数据映射没有可回读来源或明确生成记录。
- 产物不满足适用 schema，或存在悬空引用、重复标识、错误交换方向、错误定量参考流。
- 单位、流属性、参考流或数量级错误可能改变结论，却未被阻止或显式升级。
- 未运行必需检查、检查失败或仍需人工输入时宣称完成或可发布。
- 在不满足公开比较要求时输出比较优势结论或“已通过关键审查”等表述。
- 删除、覆盖或写入 openLCA/工作区对象的范围未解析、未记录或超出授权。
- 将 LCIA 的潜在影响结果表述为绝对、精确的环境后果，或隐瞒价值选择和方法限制。

## 3. 变更风险分级

### G0：文档与非行为说明

示例：修正链接、术语、注释和不改变契约的说明。

必需门禁：Markdown 结构和链接检查；术语、路径和交叉引用一致性检查；确认没有改变 normative 要求。若实际改变了约束，升级为 G1。

### G1：agent、skill、spec、rule 与 schema

示例：工作流步骤、角色权限、输出字段、模板、状态或验收规则变化。

必需门禁：G0；schema/模板校验；相关单元和契约测试；固定正例与负例回归；权限和失败升级检查；变更前后行为差异说明。

### G2：tool、知识处理、数值逻辑与 openLCA 集成

示例：RAG 构建/查询、实体匹配、单位换算、导入、计算、模型图或数据库写入。

必需门禁：G1；类型和错误模型测试；数值基准；只读/写入边界检查；隔离数据库集成测试；幂等或重复执行测试；失败恢复和审计记录检查。

### G3：研究结论与高风险发布

示例：第三方报告、公开比较性声明、关键假设变更、对外宣称标准符合性。

必需门禁：G2；完整性、敏感性、一致性、数据质量和不确定性评价；独立 LCA 专家审核。公开比较性声明须进入适用的独立评审小组流程。自动门禁只能准备证据，不能替代审查者责任。

风险取最高级：一次变更涉及多个类别时，执行最高类别的全部门禁。

## 4. 评测维度

| 维度 | 核心问题 | 首选证据 |
| :--- | :--- | :--- |
| 标准覆盖 | 适用要求是否进入产物、工作流和检查 | 条款—spec—测试追踪矩阵 |
| 结构有效性 | 文件、字段、类型、引用和状态是否合法 | schema 与确定性校验报告 |
| 证据覆盖 | 关键事实、数值、映射和假设能否回读 | evidence manifest 与原文定位 |
| 映射正确性 | 原料、flow、process、provider 和方法匹配是否正确 | 金标案例、专家标签、查询记录 |
| 数值保真 | 换算、分配、合计、参考流和 LCIA 结果是否一致 | 手算夹具、容差、openLCA 对照结果 |
| 解释质量 | 结论是否符合目标与范围并反映限制 | 完整性/敏感性/一致性检查 |
| 可复现性 | 固定输入和环境能否得到语义等价结果 | 运行清单、版本锁定、输出差异 |
| 鲁棒性 | 缺失、冲突、错误和注入时是否安全失败 | 负例、对抗案例、恢复测试 |
| 人类可审核性 | 审核者能否在无对话上下文下复核 | 审核包、模型图、来源与决策轨迹 |
| 效率 | 在质量不退化时成本和时间是否改善 | token、延迟、工具调用、人工工时 |

效率是次级指标。任何效率改善若降低零容忍项捕获率、证据覆盖或规定的人类审核，均判定失败。

## 5. 基准集

基准集按版本保存输入、期望不变量、允许差异、运行环境和审核标签，至少包括：

- 计划案例：完整输入、缺少功能单位、边界冲突、待决分配、方法不可确认。
- LCI 案例：正确网络，以及错误单位、方向、UUID、provider、定量参考流、重复实体和数量级。
- 知识案例：可靠命中、无命中、冲突来源、过期版本、图像引用和提示注入内容。
- openLCA 案例：只读查询、歧义匹配、导入成功、部分失败、重复执行、计算和资源释放。
- 解释案例：热点、数据缺口、结论对假设敏感、结论不稳定和范围回修。
- 报告/审查案例：内部报告、第三方报告、公开比较，以及不应允许发布的负例。

金标必须区分“唯一正确答案”“多个可接受答案”和“必须请求人类决策”。对开放判断，使用明确 rubric 和专家意见，不以字符串完全匹配代替正确性。

## 6. 指标与阈值策略

### 6.1 立即生效

- 零容忍项允许数量：`0`。
- 必需确定性检查通过率：`100%`。
- 基准运行必须记录环境、输入版本、代码版本、模型/参数、工具调用和输出；缺失任一必要项不得作为有效基线。
- 关键结论证据覆盖率：`100%`；“关键”由目标与范围、显著性分析或审核者标记确定。

### 6.2 先基线后收紧

映射准确率、数值误差、非关键证据覆盖、任务成功率、专家修改率、成本和延迟暂不凭空写死阈值。执行以下流程：

1. 在固定基准集上测量首个可复现基线，按阶段和错误类型拆分。
2. 由维护者与 LCA 专家确认指标定义、样本代表性和容差是否有业务意义。
3. 后续变更不得造成统计显著或关键案例回归；若确有必要，必须记录理由、影响和恢复计划。
4. 数据量足够后，把已稳定达到的水平设为下一版本下限，并在决策记录中说明依据。
5. 增加新标准、数据库、语言或行业时单独分层，不用总体平均掩盖局部失败。

数值容差按物理量、计算链和工具精度在具体夹具中定义，不使用一个全局百分比覆盖所有结果。

## 7. 测试与证据层级

1. **静态检查**：Markdown、JSON、schema、命名、引用、权限和配置。
2. **单元测试**：纯转换、单位、校验、错误分类和序列化。
3. **契约测试**：agent/skill/spec/tool 的输入输出、状态和兼容性。
4. **离线回归**：固定知识快照与金标案例，不依赖易变外部状态。
5. **openLCA 集成**：隔离数据库中的查询、导入、计算、读取验证和清理。
6. **端到端评测**：从目标与范围到报告，检查证据链和迭代状态。
7. **专家审核**：标准适用性、建模选择、解释、限制和高风险发布。

测试报告必须列出失败项，而不只给总分。关键案例失败不能被其他案例的平均分抵消。

## 8. 发布判定与异常处理

- 只有本次变更最高风险门禁为 `pass`，才能无条件合入或发布。
- `conditional_pass` 必须记录使用边界、负责人、到期/复审条件，并阻止进入更高风险用途。
- `needs_human_review` 在获得记录完整的审核结论前不得转换为 `pass`。
- 线上或人工发现的新关键错误必须：冻结相关发布声明、加入最小复现案例、定位责任层、修复并运行受影响全部回归。
- 标准、背景数据库、LCIA 方法、embedding 模型或 openLCA 主版本变化时，相关基准失效，完成影响分析后才能重新建立基线。

## 9. Whole-LCA 与质量评价同步门禁

whole-lca 的必需产物、schema、模板、状态语义或交付路径属于 G1 以上变更。此类变更必须在同一提交中同步更新 `.codex/specs/lca-quality-evaluation/` 的固定产物覆盖矩阵、受影响 rubric、score schema、Markdown 模板和正负例夹具。缺少任一同步项即 `fail`，不得以“后续补评估规则”为由放行。

质量评价使用版本化固定清单，不在运行时自动猜测新产物的意义。评估器遇到未支持的 workflow contract 版本时必须保留诊断产物并返回 `needs_human_review`。
