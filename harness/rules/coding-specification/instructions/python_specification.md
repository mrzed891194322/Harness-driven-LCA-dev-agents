# Python 脚本编写与运行规范 (Python Specification)

本文件定义了在本项目中编写与运行 Python 脚本的运行环境、范围限制、临时脚本存放以及文件操作的安全边界。

## 核心规范

为了保障项目开发环境的一致性、代码结构的整洁与系统安全性，所有 Agent 在编写和执行 Python 脚本时，必须严格遵守以下约定：

1. **Python 运行环境要求**
   - 项目中由 `uv` 管理的虚拟环境（通常位于 `.venv/`）是**唯一可用**的 Python 运行环境。
   - 严禁使用任何外部系统 Python 环境或创建其他非项目约定的虚拟环境。
   - 如果在开发或运行过程中有安装新第三方包的需求，**必须征得用户同意**后方可进行安装。

2. **编写与运行范围限制**
   - 所有 Python 脚本的编写与运行只能在 `GUI/`、`harness/tools/`、`scripts/` 与 `workspace/` 目录以内。
   - 严禁在上述约定的代码目录以外（如项目根目录直接存放、其他非代码目录）编写或执行任何 Python 脚本。

3. **临时脚本存放约定**
   - 如果是用于测试、实验或探索环境等“阅后即焚”的临时脚本，必须统一存放在 `workspace/tmp/` 目录中。
   - 严禁将临时测试脚本散落在其他任何目录中，且临时脚本应在运行及验证完毕后及时清理。

4. **文件操作安全边界**
   - 编写的 Python 脚本在运行过程中所能操作（包括读取、写入、修改、删除等）的所有文件，**仅限于本项目目录以内**。
   - 严禁在脚本中编辑或操作项目外部的任何文件（如系统临时文件、桌面文件、用户主目录下的外部文件等）。

5. **工作脚本编写与结构规范（针对 workspace/data、workspace/eval 等）**
   - **可复用原则**：编写用于数据处理、模型评估等长期工作的 Python 脚本时，必须注重代码的可复用性。进行可复用性检查时，**必须以子目录的名称（即任务名称）为线索**，排查是否已存在相似任务的脚本。若发现相关目录下已存在类似的脚本（例如曾执行过相似任务但需修改），应**优先考虑修改与完善现有脚本**，严禁无故新建重复功能的脚本。
   - **模块化结构**：针对每一个具体任务，所有相关的文件必须放在一个独立的子目录中，且**该子目录的名称即为所需开展的任务名称**。在该子目录内采用以下模块化结构：
     - `main.py`：作为脚本的**唯一主入口与总接口**，其中仅能包含唯一的 `main()` 函数。
     - `utils/` 子目录：所有具体的功能实现函数必须根据用途进行模块化拆分，存放在 `utils/` 子目录下的不同 Python 脚本中。如果包含配置文档，也可按需存放于该子目录下。
     - `README.md` 说明文件：该任务子目录内必须包含 `README.md` 解释整段代码。在后续修改/阅读代码时作为参考，且**在修改代码时，必须同步更新 README.md**。
     - **结构模板**：编写时可参考项目提供的模板目录：`harness/rules/coding-specification/templates/py_scripts/`。
