# LCA 项目计划制定规范 (LCA Planning Logic & Guidelines)

本规范定义了基于 openLCA 程序的 LCA（生命周期评估）项目方案制定阶段的核心要求与指导逻辑。

智能体在分析和制定方案时，必须严格遵守以下原则：

---

## 1. 计划制定核心要求

制定的 LCA 项目计划必须包括并体现以下四个方面：

### 1.1 基本范围定义
- **研究对象与目的**：明确评估的产品、技术或系统，阐述开展研究的目的（如识别环境热点、EPD 声明等）。
- **功能单位 (Functional Unit, FU)**：设定明确的功能单位，包含基准流量和物理单位，并阐明设定依据。
- **系统边界与截断规则**：界定生命周期边界（如 Cradle-to-Gate），说明质量/能量截断比例（如忽略贡献低于 1% 的流）及特殊排除。
- **多产出分配原则**：若存在副产品，说明分配优先级（优先选择系统扩展，其次物理或经济分配）。
- **生命周期影响评价方法**：确定 LCIA 方法（如 EF 3.1, ReCiPe 2016 Midpoint）与关注的核心指标。

### 1.2 数据与环境基础
- **数据库与连接**：确认 openLCA 连接地址、端口及已载入的数据库（如 ecoinvent, UVEK）。
- **参考资料情况**：评估已有物料清单(BOM)、技术文档等数据质量。
- **核心数据要求**：在确定数据集与 LCIA 方法时，必须尽可能参考现有数据。除了用户提供的文件输入外，还必须通过 Python IPC 接口与 openLCA 通信，获取并参考其数据库内的已有数据与现有方法。

### 1.3 模型细节方案
- 明确需要在 openLCA 中新建的流 (Flow) 与过程 (Process)。
- 明确背景数据链接（选择并说明所使用的背景数据库，如 ecoinvent 等）。
- 明确如何构建产品系统 (Product System)。
- 明确是否需要使用 Project 功能进行多情景对比分析。

### 1.4 结果的验证与解释方案
- 明确如何进行结果的验证（如完整性与一致性检查）。
- 明确结果的解释方案（如基于贡献树、桑基图等的热点分析手段）。

---

## 2. 计划制定的两种场景：新计划制定与老计划修改

根据项目的不同阶段，智能体必须区分并执行以下两种计划制定逻辑：

### 2.1 场景 A：新计划制定 (Formulating a New Plan)
- **适用条件**：当目标执行计划文件 `src/plan/execution_plan.md` 和待完善清单文件 `src/plan/todo_list.md` 尚不存在，或用户发起全新的项目计划制定时。
- **执行逻辑**：
  1. 智能体应完整且显式地读取提示词中指定的初始化需求文档及相关参考背景文档。
  2. 按照本规范 **第 1 部分（计划制定核心要求）** 建立全新的 LCA 项目计划框架并写入 `src/plan/execution_plan.md`。
  3. 将所有无法确定的信息、缺口数据和决策项汇总至新建的 `src/plan/todo_list.md` 中。

### 2.2 场景 B：老计划修改 (Modifying an Existing Plan)
- **适用条件**：当已生成的执行计划文件 `src/plan/execution_plan.md` 和待完善清单文件 `src/plan/todo_list.md` 已经存在，且用户通过交互界面或修改文件提供了具体反馈时（例如：在 `src/plan/todo_list.md` 的「用户反馈答复」区域中填写了答复，或在提示词指定的修改意见文档中填写了修改需求）。
- **执行逻辑**：
  1. **读取现有成果与用户反馈**：智能体**必须首先完整读取**已生成的 `src/plan/execution_plan.md` 和 `src/plan/todo_list.md`，并提取用户在各个 `***✍️ 用户填写内容区***` 内填写的最新反馈、答复内容、以及用户的额外修改需求。
  2. **在现有文件上进行增量修正**：**严禁随意推翻或重新生成**与用户反馈无关的已有设计。智能体必须基于用户反馈，针对性地对 `src/plan/execution_plan.md` 的对应模块进行局部精准修正与细节丰富（例如：根据用户对分配原则的反馈修改其中的分配原则描述；根据用户补充的数据修改过程/流设计等）。
  3. **更新待完善清单状态**：一旦用户对某个待完善清单条目做出了明确的答复，在修改 `src/plan/execution_plan.md` 的同时，必须将 `src/plan/todo_list.md` 中对应的条目予以更新或标记为已解决，将已解决的内容固化进执行计划中，从而实现方案的迭代收敛。

---

## 3. 待完善清单 (Todo List) 格式与生成规范

导出的待完善清单 `src/plan/todo_list.md` 必须严格、且仅能包含以下两个主要部分（一级标题）：

### 3.1 核心结构与逐条参考要求
1. **「## 1. 待完善清单内容与反馈答复」**：
   - 此部分的内容**必须完全、逐条参考** `execution_plan.md` 中留空、带中括号占位符（如 `[请在此处明确...]`）或提及存在不确定性、缺口、待决策的问题。
   - 每个问题/项必须对应为一个带编号的子标题/列表项，并包含 `***✍️ 用户填写内容区***` 段落供用户填写反馈。
   - 严禁在此部分之外生成任何额外的一级标题（如单独的数据库关联、分配决策等），所有这些疑问都必须归入此部分下。
2. **「## 2. 用户额外补充意见与文字需求」**：
   - 此部分是预留给用户的额外文字输入区，结构必须与模板一致，严禁省略。

### 3.2 严禁的行为
- 严禁生成 `## 3.` 或更多的一级标题。生成的文档必须**仅有以上两个部分**。
- 严禁编造或插入与 `execution_plan.md` 中提出的问题无关的赘余内容。

---

## 4. 多重评估自检与 Agent 循环 (Agent Loop)

在将制定的方案与待完善清单正式导出为文件后，必须建立闭环评估机制：

- **调用评估**：明确调用 `eval-executor` agent 对生成的报告文件（计划文档和待完善清单）进行质量与规范性评估。
- **验证依据**：请要求 `eval-executor` 显式读取并对照 `evaluation/self_check.md` 执行自检校验。
- **构建循环 (Agent Loop)**：如果 `eval-executor` 的评估结论认为“需要改进”，你必须根据其反馈重新调度子智能体（如 `doc-handler`）修改已生成的文件内容，随后再次提交评估。如此循环迭代，直到 `eval-executor` 认为方案达标或可交付。
