# src 内部目录规范

> **上下文提示**：如需了解项目整体结构（根目录、输入输出等），请另读同目录下的 `global-structure.md`。
>
> 本文件由 `project-convention` skill 维护，聚焦**项目核心代码（src/）内部**的目录结构、模块职责与关键约束。

---

## 1. 目录结构

```text
src/
│
├── data/                   # 数据处理代码及产生的中间数据
├── eval/                   # 评估工具（LCA 合规性检查、指标验证）
├── history/                # 历史运行记录与对话上下文存档
├── knowledge/              # RAG 知识库（由 embedding 模型动态生成与读取）
├── plan/                   # 当前执行中的内部设计方案、子计划存放
└── tmp/                    # 临时文件及 AI agent 临时脚本目录
```

---

## 2. 各目录详细约定

### 2.1 `data/` — 数据预处理与中间状态

- **职责**：处理来自于 `input/data/` 的原始输入数据。此处可以包含数据清洗和转换的脚本文件（例如 Python 脚本），以及脚本处理后生成的中间输出文件（例如提取好的标准化 LCA 清单数据）。
- **约束**：原始数据不应直接复制到此处，而是由脚本动态读取 `input/` 下的文件进行生成。

### 2.2 `eval/` — 评估验证与合规性检查

- **职责**：提供对 LCA 报告、处理流及计算结果的规范性和合规性检查工具、检查逻辑。
- **设计原则**：评估逻辑需与数据处理逻辑解耦。倾向于设计成可复用的校验函数或者规则引擎，用来独立判断最终数据或报告是否满足外部交付要求。

### 2.3 `history/` — 历史记录归档

- **职责**：存放每次系统完整运行、重要交互、或模型执行的版本记录归档。方便后续分析追溯以及多 Agent 在长周期任务之间的状态记忆共享。

### 2.4 `knowledge/` — 动态生成的 RAG 知识库

- **职责**：这个目录下放的是以 RAG（Retrieval-Augmented Generation）形式构建的知识库。它通过 embedding 模型动态生成与读取，用于供 Agent 在解决特定 LCA 问题时作为“运行时热知识”进行检索，这与 `docs/knowledge_base/` 中沉淀的基础静态规范不同。

### 2.5 `plan/` — 具体执行规划与策略

- **职责**：存放代码和逻辑层面的执行设计文档。
- **关联**：相对于 `input/plan.md` 所给出的粗粒度需求或产品级目标，`src/plan/` 里应该存放技术视角的细致执行方案和多 Agent 协同拆解后的子任务蓝图。

### 2.6 `tmp/` — 临时文件目录

- **职责**：存放运行时的所有临时缓存文件。
- **重要约束**：AI Agent 在执行过程中为了做实验、运行测试代码或探索环境等所编写的“阅后即焚”脚本（临时脚本），**必须并且只能**存放在此目录。不能将测试代码到处散落。

---

## 3. 开发执行流程推荐

1. **输入解析**：主 Agent 分析项目根目录的 `input/plan.md` 需求文档与 `input/data/` 数据。
2. **规划细化**：在 `src/plan/` 目录下将需求拆解出更具体的技术路线和执行清单。
3. **数据清洗**：在 `src/data/` 目录下编写和运行脚本，将原始数据转换提炼成后续流程能直接使用的标准化数据。
4. **知识填充与检索**：如果遇到复杂的 LCA 标准或特殊换算资料，系统会通过 embedding 模型将其构建为 RAG 知识库存放于 `src/knowledge/`，Agent 可在后续处理环节动态检索读取。
5. **合规检查**：调用 `src/eval/` 目录中的评估逻辑对结果进行检验。
6. **归档与交付**：检查无误的结果导出至根目录 `output/`；过程中重要的里程碑与上下文信息记录至 `src/history/`。
